# Keyword Research — assembling the brief

> The writer cannot produce a Google-grade post from a keyword alone. The keyword must be turned into a *brief* — SERP shape, what's actually ranking, what real searchers asked next, what audience pain looks like in their own words, what angle is open. On Scribbloo the heavy lifting is already done: the keyword fan-out has been run and a `plan/` of briefs exists. This skill tells you how to draw from that plan first, then enrich it with live SERP/PAA recon before drafting.

---

## Inputs

The pipeline accepts one of:

1. **A bare keyword**: `how to draw a rose`
2. **A keyword + content type**: `how to draw a rose` (✏️ tutorial)
3. **A plan row reference**: row #3 → reads the brief for the row's keyword + slug + content type from `plan/`
4. **A keyword + slug + content type** (full manual override)

If the input is just a keyword and the keyword has multiple plausible content types, decide it from the SERP intent (Pass 3) or match it against `plan/plan.md` for any row with that keyword.

## The primary source: Scribbloo's existing pipeline + plan

Before any web recon, **check the project's own research.** The keyword work is already done and stored in the repo. Do not re-invent it.

### The `plan/` folder (start here)

The content plan lives in `plan/` and is the first place to look for any row you've been handed:

- **`plan/plan.md`** — the master table: every planned page with its slug, type (📂 collection · ✏️ tutorial · 📝 listicle · 🛠️ tool), primary keyword, monthly volume, KD (keyword difficulty), status (🆕 new / ✅ refresh), and a link to its brief.
- **Cluster briefs** — the per-page targets, grouped by cluster:
  - `plan/how-to-draw.md` — drawing tutorials (e.g. *how to draw a rose*, *how to draw a butterfly*, *how to draw a dog*).
  - `plan/coloring-collections.md` — coloring category landings (e.g. *christmas coloring pages*, *unicorn coloring pages*, *dinosaur coloring pages*).
  - `plan/drawing-ideas.md` — listicles (e.g. *cute drawing ideas*, *easy things to draw*).
  - `plan/tools.md` — tool landings (e.g. *photo to coloring page*).
- **`plan/00-writing-guide.md`** — voice, the exact frontmatter schemas, and the per-type templates. Read it for any draft.

Each brief row already gives you the **primary keyword + volume + KD + a list of "Also target" secondary keywords**. That is the spine of your brief. Pull it verbatim — those numbers came from the real keyword tool, not a guess.

### The pipeline that produced it (`site-infra/keyword-research/`)

The plan was generated by a four-stage Python pipeline. You usually *read its outputs*; you only *re-run* it when expanding into a new cluster of keywords.

```
fanout.py    → analyze.py → report.py → build_plan.py
(DataForSEO    (clean,       (cluster    (writes the plan/
 Labs + Apify   dedupe,       summary +   briefs from
 autocomplete)  cluster,      top         MASTER_keywords.csv)
                score)        opportunities)
```

- **Engines:** **DataForSEO Labs** (related keywords, keyword suggestions, keyword ideas — real monthly volume, CPC, competition, KD, intent) and **Apify Google Autocomplete** (the long-tail suggestion fan-out). Called by `curl`/`urllib`, no MCP.
- **Credentials** already live in the repo `.env`: `DATAFORSEO_LOGIN`, `DATAFORSEO_PASSWORD`, `APIFY_TOKEN`. You do not add or invent keys — they're configured.
- **Key data files** in `site-infra/keyword-research/`:
  - `MASTER_keywords.csv` — the merged, scored, deduped keyword set (the source of truth for volume/KD).
  - `keywords_clean.csv` / `.json` — cleaned and clustered keywords.
  - `insights.json` — the opportunity list (decent volume, low KD) and the question/informational keywords (blog fodder).
  - `REPORT.md` — the human-readable cluster summary.

To look up demand for a keyword that isn't already in a brief, grep `MASTER_keywords.csv` rather than guessing a number. To expand the universe (a brand-new theme), re-run `fanout.py → analyze.py → report.py → build_plan.py` with the new seeds added to `fanout.py`'s seed lists.

## How the live recon is run (the complementary method)

The plan tells you *what* to write and *the demand behind it*. It does not capture today's live SERP — the featured snippet, the exact PAA wording, how the current top 3 open. For that you run a hand recon with **WebSearch** and **WebFetch** — the same tools any reader has. There is nothing to install. The rigor comes from *how* you read the SERP, not from an API.

Four passes. Passes 1–3 cover the SERP, the related space, and the competitors; Pass 4 captures audience voice; the synthesis at the end is something you write, not something an API returns.

### Pass 1 — Study the live SERP (what's actually ranking)

Run the keyword as a WebSearch exactly as a user would type it. Read the result page the way Google presents it and capture every signal:

- **The top 10 organic results** — URL, title, and the snippet Google chose. These are the pages you have to beat.
- **The featured snippet** (if present) — which URL holds it, its shape (paragraph / numbered list / bulleted list / definition), and its exact text. This is your snippet target; you are trying to take it.
- **People Also Ask (PAA)** — capture the questions verbatim. Expand a couple and note the answers Google is surfacing. PAA is the richest source of sub-questions to cover as H2s or in an FAQ block.
- **Related searches** ("people also search for" at the bottom) — capture the terms verbatim. These become H2 candidates, internal-link targets, or sibling-post ideas.
- **SERP feature mix** — is there an image pack, a video block, a "things to know" carousel? A SERP dominated by image packs and Pinterest grids (common for *coloring pages* and *cute drawing ideas*) means the post must earn the image-pack click and lead with the printable/step images.

Run 2–4 variant queries to widen the picture. For `how to draw a rose`, also search `how to draw a rose easy`, `how to draw a simple rose`, and `rose drawing step by step` — the PAA and related-searches sets differ across variants and together map the topic.

### Pass 2 — Map the related-keyword space (volume by proxy, or pull the real number)

On Scribbloo you usually have the **real number** already — the brief lists volume and KD, and `MASTER_keywords.csv` holds the rest. Pull from there first. When a variant isn't in the data, infer demand from signals you *can* see:

- **Autocomplete** — start typing the seed into the search box and record the suggestions. Google orders autocomplete roughly by popularity, so `how to draw a` → `rose`, `dog`, `cat`, `butterfly` tells you the dominant subjects. (This is the same Apify autocomplete signal the pipeline harvested at scale.)
- **"People also search for"** from Pass 1 — already a ranked-ish related set.
- **Modifier families** — group the related terms into families the post should cover: the *easy/simple* family (`how to draw a rose easy`, `simple rose drawing`), the *step-by-step* family (`rose drawing step by step`, `how to draw a rose for beginners`), the *style* family (`cute rose drawing`, `realistic rose drawing`), the *coloring* family (`rose coloring page`).
- **Trend sanity check** — if you have any doubt the term has demand, the brief's volume figure settles it; if it isn't in the data, a quick WebSearch plus "Google Trends" confirms it isn't dead or purely seasonal (e.g. *christmas coloring pages* spikes in Q4).

Mark volume as the brief's figure where you have it, otherwise "inferred: high / medium / low" from autocomplete position and PAA presence. The brief needs *relative* priority (which modifiers to cover, in what order) plus the real number where the plan supplies it.

### Pass 3 — Read the top 3 competitors (find the open angle)

Use WebFetch on the top 3 organic URLs from Pass 1 and read what they actually say. For each, record:

- **URL** and **H1**
- **The H2 list** — what sections they cover, in order
- **Rough word count** — how long the post is
- **How they open** — do they answer in the first paragraph, or bury the steps behind 600 words of preamble?
- **Whether they show the steps clearly** — is there a clean numbered step-by-step with images, or just a wall of prose?
- **Whether they offer a printable / coloring page** — or leave the reader with nothing to take away?

This is what shapes the angle. The pattern you are hunting for is *what all three do badly*:

- If all three open with "In this article we will explore…" — yours opens with the encouraging hook and the first shape to draw.
- If all three have an H2 titled "Introduction" or "History of the rose" — yours leads with the materials list and step 1.
- If all three bury the actual steps behind 800 words of filler — yours puts the numbered steps and an image in the first screen.
- If all three are 1,200 words of identical prose — yours either goes tighter (a clean answer-first tutorial that wins the snippet) or substantially deeper (a fuller guide with cute/realistic variations and a matching coloring page).
- If all three make the reader hunt for something to color afterward — yours links straight to the matching coloring collection (e.g. `/coloring-pages/flowers` once it exists, or the closest live collection).

Also classify the **dominant intent** from the SERP makeup: if the top 10 are step-by-step tutorials, the intent is *learn to draw it* (✏️ tutorial); if they are image grids of printables, the intent is *get pages to print* (📂 collection); if they are "X ideas" listicles, the intent is *get inspiration* (📝 listicle). The intent decides the content type and the skeleton.

### Pass 4 — Capture audience voice (optional but high-signal)

For pillar and listicle topics, find how real people phrase the question and where they get stuck. WebSearch the keyword with a community qualifier and read the threads with WebFetch:

- `how to draw a rose reddit`
- `drawing for beginners site:reddit.com`
- `<keyword> tips` on a community
- `<keyword> quora`

For drawing topics, try `r/learnart`, `r/drawing`, `r/IDAP` (I Drew A Picture); for kids' activities and coloring, try `r/Teachers`, `r/Parenting`, and craft/homeschool communities; for color/medium questions, art-supply community threads are gold.

Stash:

- **5–10 verbatim reader questions** (their exact phrasing — do not paraphrase)
- **3–5 verbatim pain-point quotes** ("I follow the steps but my rose always looks like a cabbage — I can't get the petal layering to look right…")
- **Common mistakes** — especially the *technique errors* readers actually make: starting with detail instead of basic shapes, pressing too hard with the pencil so they can't erase guidelines, drawing every petal the same size, coloring outside the lines because the printout was too light/thin.
- **Rules of thumb the audience trusts** — and whether they are actually sound technique.

This is the highest-signal source of *real audience voice*. SERP research gives you keywords; communities give you the language readers actually use and the exact spot where they get confused — which is where your worked steps and tips box should focus.

### Synthesis — you write the brief

With Passes 1–4 (and the plan row) in hand, compress the research into the brief yourself (template below). There is no synthesis API call — you are the synthesizer. Read across the plan row and the four passes and answer: what are the reader intents, what is the open angle, which facts/techniques must be exactly right, which sources to cite, which PAA to cover, what to link to internally.

## The assembled brief (output of this pipeline)

After the four passes, you have assembled a brief covering:

```
## TARGET QUERY
"<keyword>"  (volume: <from plan/MASTER_keywords.csv, or inferred high|medium|low> · KD <from plan>)

## CONTENT TYPE
<✏️ tutorial | 📝 listicle | 📂 collection/category | 🛠️ tool>
(from the plan row, or inferred from SERP intent in Pass 3)

## URL / SLUG
<slug from the plan row — keep it; don't invent a new one>
content/blog/<slug>.mdx  (tutorial/listicle)  ·  content/categories/<slug>.mdx  (collection)

## RELATED COLORING COLLECTION / PAGES
relatedCategories: [<slugs>]  ·  relatedPages: [<slugs>]
The collection or sibling tutorials this post cross-links to. If the target doesn't exist yet,
note "planned — not yet live" so the writer doesn't link a 404.

## SERP shape
- Featured snippet currently: <yes/no, shape, holding URL>
- PAA questions: [<5-10 verbatim>]
- Top 3 ranking: <URLs + H2 lists + word counts + opening style + clear-steps? + offers-printable?>
- Related searches / autocomplete: [<terms>]
- Dominant intent: <learn-to-draw | get-printables | get-inspiration>

## OPEN ANGLE
<2-3 sentences on what the top 3 are missing — specific, not "more comprehensive">

## AUDIENCE VOICE (from communities)
- Real questions readers ask: [<5-10 verbatim>]
- Pain points: [<verbatim quotes>]
- Common mistakes: [<bullets — including the technique errors beginners actually make>]

## PRIMARY SOURCES
- [<URL, outlet, relevance — art-instruction reference / museum / established technique guide preferred>] × 3-8

## VERIFIED FACTS / TECHNIQUES TO FEATURE
- <fact or technique step> — <materials/terms with correct vocabulary> — <source: art-instruction reference / museum / established guide> — <verification confidence>
- × per content-type need

## TRADEMARK FLAG (licensed characters)
- Is the subject a licensed character (Hello Kitty, Pokémon, Sonic, Bluey…)? <yes/no>
- If yes: use fan-style phrasing ("fan-art style", "inspired by"), add the light disclaimer,
  prefer a generic equivalent where intent allows. (See research-and-citation-skill.md §trademark.)

## H2 PLAN (per content type)
<H2 list per the content type's template in plan/00-writing-guide.md §8>

## INTERNAL LINK TARGETS
<the matching coloring collection + 3-5 sibling slugs + 1 pillar slug, drawn from the existing
content/blog/ and content/categories/ directories and the plan>

## FAQ QUESTIONS (PAA capture)
- <PAA question 1> — <40-60 word answer plan>
- × 2-4

## FEATURED SNIPPET TARGET
- query: <the snippet target>
- shape: paragraph | numbered list | bulleted list | definition
  (NOTE: GFM pipe tables are NOT guaranteed to render in post bodies — prefer prose/list shapes;
   the styled blockquote is the answer box.)
- answer: <40-60 word answer plan — this becomes the opening blockquote>
```

This brief is what the writer receives. It does not exist as a saved file by default — it lives in context for the duration of the write. You can choose to save it as `briefs/<slug>.md` for audit / review purposes.

## When to save the brief vs run inline

Two modes:

### Inline (default)
The brief is assembled, the writer drafts immediately, the brief is discarded after the post ships. Faster, simpler.

### Saved
The brief is saved to `briefs/<slug>.md` before drafting. Useful for:

- Topics where you want to review the brief before drafting
- Posts that may need re-drafting later with the same research base
- Topics with heavy trust risk (a licensed character, an art-history claim, a multi-step technique) where extra review is warranted

The default is inline.

## When to skip or trim the pipeline

The pipeline assembles a brief from scratch. Skip or trim the relevant passes if:

- **The plan brief is already rich** — `plan/plan.md` + the cluster brief give you primary + secondary keywords + volume + KD + slug. For a straightforward ✏️ tutorial or 📂 collection you may skip Pass 2 entirely and lean on Pass 1 (PAA + snippet) and Pass 3 (competitors).
- **The brief already exists** (operator-provided at `briefs/<slug>.md`) — skip Passes 1–4, jump straight to drafting.
- **The keyword is a simple, well-understood subject** (e.g. *how to draw a heart*, where the steps are standard and the SERP is settled) — run Pass 1 (snippet + PAA) and Pass 3 (open angle) only; skip Pass 4.
- **The subject is a licensed character** — do NOT skip the trademark flag; that check is mandatory regardless of which passes you trim.

## Quality bar

A bad brief produces a bad post no matter how good the writer is. The brief is the load-bearing artifact. Specifically:

- **The open angle must be specific.** "More comprehensive than competitors" is not an angle. "Every competitor makes you scroll past 600 words of rose history before the first stroke; we put the basic-shape step and an image in the first screen, and we show the petal-layering trick the top three all skip" is an angle.
- **The featured facts and techniques must be correct and verifiable, not guessed.** The writer should not assert that "graphite is made from lead" (it isn't — it's carbon) or invent an art-history claim. Steps must actually work when followed. Where a factual claim appears (a medium's behavior, a color-theory term, an art-history note), it traces to a real source. (See `accuracy-and-trust-skill.md` — the trust gate.)
- **The domain facts must trace to a real source.** Technique conventions, material properties, color-theory definitions, art-history notes — never asserted from memory; cited to an established art-instruction reference, a museum, or a recognized technique guide. Where practice genuinely varies (e.g. a cute vs realistic approach, different paper for markers vs watercolor), give the distinction, not false certainty.
- **Licensed characters are flagged.** Any subject that is a trademarked character is marked in the brief with the fan-style/disclaimer handling, so the writer never implies official status.
- **The audience voice should be verbatim.** Reader quotes from communities aren't paraphrased — they're stored as exact strings.
- **The internal-link targets must exist.** Before listing `/coloring-pages/flowers` or `/blog/how-to-draw-a-tulip` as a link target, confirm the collection exists in `content/categories/` or the post exists in `content/blog/` (or is in `plan/plan.md` as planned). Never link a 404. Use correct material/terminology vocabulary throughout (HB vs 2B, lb vs GSM) — see `materials-and-terminology-skill.md`.

## Pre-write checklist (run this on the assembled brief)

- [ ] Target query is specific and matches a real search intent
- [ ] Content type is decided (✏️ tutorial / 📝 listicle / 📂 collection / 🛠️ tool) and matches the URL path
- [ ] Slug taken from the plan row (not invented); matching collection/sibling slugs identified (or marked "planned — not yet live")
- [ ] Volume + KD pulled from the plan brief / `MASTER_keywords.csv` where available
- [ ] At least 3 competitor pages were read (Pass 3)
- [ ] At least 3 primary sources are identified (art-instruction reference / museum / recognized technique guide preferred)
- [ ] The featured facts/techniques are correct and traced to a citable source
- [ ] Licensed-character subjects are flagged with fan-style + disclaimer handling
- [ ] H2 plan matches the content type's template in `plan/00-writing-guide.md`
- [ ] Internal-link targets are real slugs (verified against `content/blog/`, `content/categories/`, and `plan/plan.md`)
- [ ] PAA questions are real (lifted from Pass 1, not invented)
- [ ] Featured-snippet target is set, with a render-safe shape (no GFM table in the body)

## Re-running or extending the pipeline (only when expanding the keyword universe)

The `plan/` folder is already built. You re-run the pipeline only to expand into new themes — not for every post. The pipeline is checked into the repo and uses the credentials already in `.env`; you run it, you don't reconfigure it.

To add a new cluster (say a brand-new coloring theme), add seeds and re-run the four stages from `site-infra/keyword-research/`:

### Stage 1 — fan-out (DataForSEO Labs)

Add your new head terms to the `RELATED_SEEDS`, `SUGGESTION_SEEDS`, and `IDEA_SEEDS` lists in `fanout.py`, then:

```bash
python site-infra/keyword-research/fanout.py
```

This walks DataForSEO Labs `related_keywords` (depth fan-out, like a human clicking "related searches"), `keyword_suggestions` (long-tail), and `keyword_ideas` (category-adjacent), merging real volume, CPC, competition, KD, and intent into `keywords_all.csv` / `.json`.

### Stage 2 — clean, dedupe, cluster, score

```bash
python site-infra/keyword-research/analyze.py
```

Drops off-topic noise, collapses near-variants (page/pages, idea/ideas), assigns each keyword to a thematic cluster (animals, unicorn/fantasy, seasonal, audience, format, flowers/nature, drawing ideas, how-to), and surfaces opportunities (volume ≥ 500, KD ≤ 20) and question/informational keywords. Writes `keywords_clean.csv` / `.json` and `insights.json`.

### Stage 3 — report

```bash
python site-infra/keyword-research/report.py
```

Writes `REPORT.md` — the human-readable cluster summary and top opportunities.

### Stage 4 — build the plan

```bash
python site-infra/keyword-research/build_plan.py
```

Turns `MASTER_keywords.csv` into the `plan/` briefs (`plan.md` + cluster files), matching against existing `content/**/*.mdx` to mark rows ✅ exists vs 🆕 new, and keeping multiword entities (Hello Kitty, Pokémon) intact through tokenization.

The Apify Google Autocomplete data (`apify_autocomplete.json` and friends) feeds the long-tail fan-out; it's harvested separately via the Apify actor using `APIFY_TOKEN`. The manual WebSearch/WebFetch recon above costs nothing beyond the tools you already have and is the right tool for enriching a *single* row before you draft.
